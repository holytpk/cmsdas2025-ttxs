    Histograms and Plotting basics

    Output formats

    Now that we have all our samples and the lumi weights set up, let's take a look at what Pepper outputs. You can run in debug mode for starters.

    By default, Pepper will save hists in the hist.Hist format, allowing for easy integration with the scikit-hep ecosystem. However, it is also possible to export to ROOT files+histograms. This is done by pointing one of the Pepper scripts to your Pepper state file:

    python scripts/export_hists_from_state.py <input_path>/state.coffea root <output_path>

    These outputs can then be viewed via, for example, a TBrowser.

    To-do

    try taking a look at the outputs, either with a ROOT file viewer or natively in python. What do you see? Does it make sense?

    Plotting the outputs

    Ultimately, we want to see information about all the different samples displayed together. To do this, we use Pepper's plotting script, pointed at the output file hists.json which compiles information about the different saved histograms:

    python3 scripts/plot_histograms.py <config file> <processor_output_path>/hists/hists.json

    You made need to add plotting information for the background samples we added ourselves earlier--choose a good plotting color for them!

    Try running the plotting script on your outputs. What do they look like? Are the different samples visible?

    Adding more histograms

    Before we run on the full sample statistics, let's see how to add our own histograms. Histograms are defined in their own config file, located in the inputs/ directory, which the main config file with point to. Actually, there can be more than one! It often helps to have a few different sets of hist configs for different scenarios. The number of histograms will directly impact on the use of computing resources, so we might not want to generate all of the histograms all of the time.

    let's look at the first histogram in the config to see how they are defined: 

    "1st_lep_pt":
{
    "bins": [
        {
            "name": "pt",
            "label": "Leading lepton $p_{{\\mathrm{{T}}}}$ (GeV)",
            "n_or_arr": 20,
            "lo": 0,
            "hi": 200
        }
    ],
    "fill": {
        "pt": [
            "Lepton",
            "pt",
            {"leading": 1}
        ]
    },
    "step_requirement": "cut:Req lep p_T"
},

    This tells Pepper to save a histogram called 1st_lep_pt in the output file. in the "bins" dict, we provide one or more histogram axes, each with a name, a nice label for displaying on plots, and a binning. the binning runs from lo to hi, with n_or_arr providing either a number for the quantity of uniform bins, or directly giving an array of bin edges if non-uniform binning is desired.

    Next, we tell Pepper what to fill this histogram with. In this case, "pt": tells us we are filling the axis named above, with data taken from the "pt" column of the data["Lepton"] array as defined in the processor. We then can pass a dict that explains to the histogrammer how to choose which values from the array are included--using the function "leading" lets us sort by pT and pick the first value.

    Lastly, Pepper processes events via a series of cuts, defined in the processor. If you only want a certain histogram to be stored after a certain point in the cut flow, you can specify this by setting the parameter "step_requirement".

    Plots with full sample statistics

    Let's try one run on the full sample statistics, getting rid of the -d flag when running the processor. This is probably too lengthy to be done on your LXPLUS machine, so we will need to submit to condor. To save resources, one person can do this and share their output path!

    In order to submit to condor, one needs to add the following flags:

    • -c <N>: submit processor 'chunks' as condor jobs, spawning jobs at a time. (500 should be reasonable)
    • -i <environment> environment for condor jobs to source (use your customized version of environment.sh which should set the PYTHONPATH variable)

    It's a good idea to first run once passing the -d flag alongside these, and doing a test-run of any setup you want to send to condor. This will send a small number of jobs to condor, processing a single 'chunk' of each sample. This way, you can avoid accidentally sending many jobs which all fail.
