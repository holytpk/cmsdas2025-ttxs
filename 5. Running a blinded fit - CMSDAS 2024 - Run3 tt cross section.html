    Performing a blinded measurement

    In most analyses, it is typical to have some sort of blinding procedure in place. Analyzers will run their fit on Asimov/Simulated data and perform a number of studies before looking at data in the signal region. When you first fit on the real data, this is called unblinding. In most cases, you want to be confident in your setup before doing this.

    We will start with a "blinded" fit.

    Switching to CMSSW environment to run combine

    We will now perform the fit on the datacards we have generated. To do this, you'll want to start a new LXPLUS session and source your CMSSW/combine environment.

    To do this, you can go to your CMSSW directory, enter the src dir, and run cmsenv.

    If you want to set up a bash script that does this for you with a single command, you can create something like the following:

    setup_combine.sh: 

    #!/bin/bash
source /cvmfs/cms.cern.ch/cmsset_default.sh
cd <PATH/TO/CMSSW>/src/
eval `scramv1 runtime -sh`
cd -
    Then running this shell script will have the same effect. to verify your environment is ok, you can try running combine -h and combineTool.py -h

    Converting datacards to workspaces

    Combine takes datacards as inputs, but has an intermediate format called a workspace stored in a root file.

    Let's get ready to do a fit.

    To-do

    Go to the directory with your datacards. Use the combineTool to create a workspace in a new directory, first_fit:

    mkdir first_fit
combineTool.py -M T2W -i all.txt -o first_fit/workspace_all.root

    Now, go to the new directory with the workspace and we will be ready to run our first fit.

    Running a fit

    The first thing we usually do in a CMS measurement is to fit on the Asimov dataset. This is just a fancy name for the expected dataset based on simulation. It tells us how our fit should look if our recorded data exactly matches our simulation.

    In many cases, analyses are kept blinded until they are very mature, meaning they can only fit on Asimov data or on certain regions of their event space where they do not derive their main sensitivity. This is to prevent the bad scientific practice of altering a measurement strategy to get a desired result. In principle, one should generally decide on the fitting methods before measuring the data, perform rigorous vetting of the fit techniques, and accept the result that is given.

    We will use a maximum likelihood fit based via the FitDiagnostics in combine. Specifying Asimov data should be used instead of real data is done by passing the option -t -1. Because we are running a fit in which signal is an expected standard model process, we should also include the option --expectSignal=1. Let's run a very simple fit

    Info

    Combine, by default, fits a parameter of interest r. This is generally interpreted as a signal strength modifier--for rates of standard model processes, the expectation is r=1. In BSM searches, one could look for a process where the null hypothesis is r=0.

    One can change the underlying model in combine to fit other more complicated parameters, but this basic case suits our needs for a cross section measurement.

    To-do

    Try running the simple fit command: 

    combine -M FitDiagnostics -d workspace_all.root -t -1 --expectSignal=1
    What does the result tell you? An output root file has been created with the name fitDianosticsTest.root. Take a look at this file, can you make sense of anything in it?

    Let's run again combine -h and see that there are a massive amount of flags which can be provided at runtime. They are sorted by method, so let's skim through the general ones and those pertaining to FitDiagnostics

    Let's run the fit again with some more options provided to combine:

    • --saveNLL --saveNormalizations --saveShapes --saveWithUncertainties : this will more information in the output root file.

    Impact plots: the CMS standard

    One of the main ways CMS analyses understand and evaluate the likelihood fit result is by looking at plots of the nuisance parameter impacts and pulls.

    These show us how much each nuisance parameter affects the fit result (impacts), what the best-fit values for nuisance parameters are (pulls), and whether the pre-fit uncertainties we place on these nuisances are constrained in the fit (uncertainty bands on pulls).

    In CMS, there is a very standardized way of plotting this information with a few methods from combineTool.py.

    To-do

    Let's generate these plots for a fit on Asimov data. This is done in 4 steps:

    1. Run an initial fit and save the output including all post-fit nuisance parameter values in a workspace: 
      combineTool.py -M Impacts -d workspace_all.root -t -1 --expectSignal=1 -m 125 --doInitialFit
    2. Run several fits, one with each nuisance parameter varied up and down from it's best-fit value by its post-fit uncertainty. 
      combineTool.py -M Impacts -d workspace_all.root --doFits -t -1 --expectSignal=1 -m 125 --parallel 20
    3. Collect the information from all of the fits into a .json file 
      combineTool.py -m 125 -M Impacts -o impacts_asimov.json -d workspace_all.root
    4. Make a nice plot using the json file. 
      plotImpacts.py -i impacts_asimov.json -o impacts_asimov
      Finally, take a look at the newly-created file impacts_asimov.pdf. Which systematic uncertainties have the highest expected impact on the POI? Are any of them expected to be constrained by the fit?
